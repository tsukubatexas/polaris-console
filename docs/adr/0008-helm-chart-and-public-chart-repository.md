# ADR 0008: Helm Chart and Public Chart Repository

## Status

Accepted

## Context

Polaris Console needs to be deployable as a real Kubernetes application, not only as a local Docker or development server. The deployment path must preserve the core security boundary of the project: the browser must never receive Polaris tokens or OAuth client secrets, and the backend must not become an unrestricted proxy to arbitrary hosts.

## Decision

We ship a Helm chart under `charts/polaris-console` and publish packaged charts to a public `gh-pages` chart repository.

The chart is intentionally fail-closed:

- `config.allowedTargetHosts` is required unless `config.allowAnyTargetHost=true` is set for local development.
- `cookieSecure` defaults to `true`.
- TLS verification remains enabled by default.
- The application runs as non-root UID/GID `10001`.
- Privilege escalation is disabled, Linux capabilities are dropped, and `RuntimeDefault` seccomp is used.
- The root filesystem is read-only with only `/tmp` mounted as an `emptyDir`.
- The backend session secret is sourced from a Kubernetes Secret, either user-provided or generated by Helm.
- A NetworkPolicy is rendered by default; production operators must add explicit egress for Polaris, OAuth, DNS, and observability.

The agentic update loop now runs Helm chart checks through `scripts/run_checks.sh`, so generated Polaris API changes and chart security regressions are validated in the same repair loop.

## Consequences

The chart requires more configuration than a permissive demo chart, but the default behavior is safer for public use. Local development remains possible with `values-dev.yaml`.

Publishing is handled by the `helm` workflow. It packages the chart, generates `index.yaml`, and pushes the public chart repository to the `gh-pages` branch.
